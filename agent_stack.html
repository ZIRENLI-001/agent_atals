内容创作 —— 从灵感生成到批量发布的创作流水线
典型场景：短视频脚本生成、图文内容生产、素材理解、视频字幕摘要、批量改写
关键能力：多模态理解、长文档 / 视频素材处理、工作流编排、审校和版本管理
推荐组合：Next.js + FastAPI + LangGraph + PostgreSQL + Qdrant / pgvector + WhisperX + GPT-5.4 / Qwen2.5-VL / GLM-4.5V
为什么：创作类产品更像“多模态 workflow”，不是单次问答；需要把脚本、素材、审校串起来
阶段推荐
0-1

先做脚本生成、字幕摘要、素材标签这类单点能力。推荐 `Next.js + FastAPI + GPT-5.4-mini + WhisperX + PostgreSQL`。

成长期

补工作流编排、素材库检索、审校台和版本管理。引入 `LangGraph + pgvector / Qdrant + Langfuse`。

大规模

做批量任务队列、视频理解 pipeline、成本路由和内容安全审核。补 `Temporal / BullMQ + LiteLLM + ClickHouse`。

代表产品 / 案例
Runway · 视频生成工作台 Jasper · 内容营销写作 Canva Magic Studio · 创意工作流 CapCut / 剪映 · 素材理解与字幕
落地建议：先把流程拆成“素材理解 → 选题 / 大纲 → 生成 → 审校 → 发布”五段，每段单独可观测。视频 / 音频素材要先经过 ASR 和结构化切块，别直接把长视频喂给模型。
生活服务 —— 到家服务、预约和客服协同
典型场景：家政 / 到修 / 医美 / 门店服务预约，智能客服、工单创建、日程协调、电话回访
关键能力：语音交互、CRM / 日历 / 工单系统集成、地理位置和时段调度
推荐组合：Next.js / 小程序前端 + FastAPI / Hono + Auth.js / Clerk + Realtime API + Composio + PostgreSQL + Redis
为什么：生活服务的核心不在“模型多聪明”，而在“能不能把预约和履约流程接顺”
阶段推荐
0-1

先做客服问答、预约意图识别和 FAQ 助手。用 `小程序 / Web + FastAPI + PostgreSQL + Redis` 就够。

成长期

接 CRM、工单、门店排班、语音客服。补 `Realtime API / LiveKit + Composio + Langfuse`。

大规模

做多城市、多门店调度、智能外呼、质检和运营看板。补 `Kafka + ClickHouse + OTel + Grafana`。

代表产品 / 案例
美团 / 饿了么 · 本地生活履约 京东到家 · 到家服务协调 Sierra / Decagon · 客服自动化
落地建议：这一类最适合先从客服和预约助理切入。优先接通日历、工单、门店系统，语音链路一定要做好中断、转人工和审计留痕。
电商 —— 导购、客服、售后和商家运营
典型场景：商品问答、智能导购、售后处理、退款政策解答、商家数据助手
关键能力：商品知识 RAG、订单和库存系统工具调用、用户画像、推荐与风控
推荐组合：Next.js + FastAPI + LangGraph / PydanticAI + PostgreSQL + Redis + Qdrant + Composio / 内部 OMS 工具层 + Langfuse
为什么：电商 Agent 需要同时理解结构化商品数据和半结构化政策文档，还要安全调用订单工具
阶段推荐
0-1

先做商品问答和售后政策助手。用 `Next.js + FastAPI + pgvector / Qdrant + Langfuse`。

成长期

补订单工具调用、售后工单、商家运营 Copilot。加入 `LangGraph / PydanticAI + Redis + 工具权限层`。

大规模

促销峰值下做缓存、模型路由、风控和观测。补 `LiteLLM + Kafka + ClickHouse + OTel`。

代表产品 / 案例
淘宝 / 天猫 · 导购与客服 Shopify Sidekick · 商家运营 Copilot Amazon Rufus · 商品问答助手
落地建议：把能力拆成三条线：商品知识问答、订单类工具调用、售后规则判断。促销期要单独做限流和缓存，别让所有问题都直接打模型。
供应链与物流 —— 异常处理和流程协同型 Agent
典型场景：异常物流单处理、ETA 解释、仓配协同、采购 / 运输单证理解、承运商沟通
关键能力：多步骤 workflow、文档解析、事件流处理、权限与审计、跨系统协同
推荐组合：FastAPI + LangGraph + Temporal / BullMQ + PostgreSQL + Redis + Kafka / Redis Streams + MinerU + ClickHouse + OpenTelemetry
为什么：供应链问题不是单轮聊天，而是“多系统、多角色、多状态”的长链路流程问题
阶段推荐
0-1

先做异常单解释、ETA 问答和单证识别。推荐 `FastAPI + PostgreSQL + Redis + MinerU`。

成长期

补跨系统 workflow、队列、消息流和审计。加入 `LangGraph + Temporal / BullMQ + Kafka`。

大规模

做全链路运营看板、承运商协同和事件回放。补 `ClickHouse + OTel + Grafana + 策略权限层`。

代表产品 / 案例
Flexport · 物流协同平台 FourKites · 物流可视化 京东物流 / 顺丰科技 · 异常与调度系统
落地建议：优先做“异常单解释 + 下一步建议”，再做自动执行。文档类场景要先把提单、发票、签收单等结构化，不然下游决策会很脆弱。
打车 / 出行 —— 低延迟事件流和实时语音优先
典型场景：司机 / 乘客客服、派单解释、路线与费用问答、语音车载助手、安全事件处理
关键能力：低延迟实时链路、位置服务、事件流、语音中断、人工接管
推荐组合：React / Next.js + Go 接入层 + FastAPI AI 服务层 + Redis + Kafka + Realtime API / LiveKit + OpenTelemetry + Grafana
为什么：打车类业务对延迟、稳定性和状态一致性极敏感，AI 服务不能直接顶在所有核心链路上
阶段推荐
0-1

先做客服解释、订单状态问答、司机助手。推荐 `Go 接入层 + FastAPI AI 服务 + Redis`。

成长期

补实时语音、位置事件流、风险策略和人工接管。加入 `Realtime API / LiveKit + Kafka + OTel`。

大规模

把成本、延迟、地区差异和多模型路由纳入平台治理。补 `LiteLLM + Grafana + ClickHouse`。

代表产品 / 案例
Uber / 滴滴 · 实时出行调度 Waymo / Cruise · 车载语音交互 Sierra · 客服自动化
落地建议：AI 最适合先落在“解释、客服、辅助决策”，不适合一开始就接管实时派单核心。位置和价格相关问题要强约束，不能让模型自由发挥。
酒旅 —— 搜索、组合推荐和复杂规则问答
典型场景：行程规划、机酒套餐推荐、签改退规则问答、售后客服、目的地内容生成
关键能力：搜索比价、规则 / 政策 RAG、多语言、工具编排、行程状态管理
推荐组合：Next.js + FastAPI + LangGraph + PostgreSQL + Qdrant + LiteLLM + Exa / 搜索接口 + 多模态文档理解 + Langfuse
为什么：酒旅的难点是规则复杂、渠道多、价格变化快，AI 需要和搜索 / 预订系统紧耦合
阶段推荐
0-1

先做规则问答、目的地推荐和简单行程建议。推荐 `Next.js + FastAPI + Qdrant + 搜索接口`。

成长期

补多渠道搜索、行程状态和售后工具调用。加入 `LangGraph + LiteLLM + Langfuse`。

大规模

做实时价格路由、国际化、多语言客服和成本治理。补 `多模型路由 + 事件流 + ClickHouse`。

代表产品 / 案例
Booking / Expedia · 搜索与政策问答 携程 / 飞猪 · 行程与售后 Perplexity Comet 风格行程规划助手
落地建议：把“内容推荐”和“交易执行”分成两层。推荐层可以更开放，交易层必须严格工具化、可回滚、可审计。

TypeScript —— Agent 开发为什么值得
优势
+LLM tool_use schema 可从 Zod 类型自动生成，不再手写两份
+流式 SSE 数据有类型推导，防止字段拼错
+大型 Agent 工程重构安全，IDE 提示完备
+Vercel AI SDK、LangChain.js、Mastra 原生 TS
+前后端 monorepo 可共享同一份类型定义
注意
-配置成本（tsconfig、构建工具链）
-类型复杂时调试门槛高于纯 JS
-运行时仍是 JS，类型不等于运行时校验（需配 Zod）
最佳实践： 用 Zod 作为唯一真相来源 → z.infer<typeof Schema> 得到 TS 类型，zodToJsonSchema() 直接生成 LLM 工具描述，前后端零重复定义。

折中方案：JSDoc + // @ts-check 获得基础 IDE 提示

React —— Agent 前端核心地位
优势
+生态最大，绝大多数 AI SDK 优先支持 React（Vercel AI SDK useChat、useCompletion）
+Suspense + useTransition 天然处理 LLM 流式渲染的 loading 状态
+Server Components 可在服务端 pipe LLM 流，减少客户端 JS 体积
+丰富的工具调用结果渲染生态（代码高亮、Markdown、图表）
注意
-单独用 React 需手动搭配 Vite/Next.js，配置项多
-多余渲染问题在高频 token 更新时需用 useRef + flushSync 优化
-Server Components 心智模型复杂，Server/Client 边界易混淆
Agent 流式渲染最佳实践： useChat（Vercel AI SDK）内置 token 追加逻辑，避免手动 setState 触发过多重渲染。工具调用结果用独立子组件渲染，memo 隔离。
Next.js App Router —— 流式 Agent 产品
Route Handler：直接处理 SSE，无需额外 BFF 层
Server Actions：减少接口层 RTT，表单类 Agent 操作极简
RSC streaming：服务端直接 pipe LLM 输出到客户端
AI SDK：streamText / streamUI 开箱即用
部署：Vercel 零配置；自部署需 Node.js runtime
缺点：App Router 学习曲线陡峭
Nuxt 3 —— 适用边界
优点：Vue 3 Composition API + TS，国内社区活跃
缺点：AI SDK 生态薄，流式需自行封装
适合：已有 Vue 栈，Agent 管理后台 / 配置界面
不适合：需要深度 LLM 流集成的对话前端
SvelteKit —— 性能敏感嵌入场景
优点：编译产物极小，无运行时 VDOM，响应极快
流式：内置 load 函数天然支持流式 fetch
缺点：LLM 工具链近乎空白，团队招聘困难
定位：性能敏感型嵌入页、非主流 Agent 产品
Remix —— 内容型 Agent 产品
优点：Defer API 流式数据，Loader/Action 清晰
优点：渐进增强，无 JS 也可降级使用
缺点：AI SDK 集成不如 Next.js 丰富
适合：内容型 + 表单型 Agent，非纯对话场景
React Native + Expo —— 移动 Agent
优点：与 React Web 共享 TS 类型，逻辑层代码复用
Expo Router：文件路由，接近 Next.js 体验
流式：fetch SSE 在 RN 支持较好，EventSource 需 polyfill
离线：配合 SQLite (expo-sqlite) 可做本地 Agent 缓存
Vite + React SPA —— 嵌入 / 插件场景
场景：Chrome 扩展、Electron 嵌入、内嵌 Agent 侧边栏
优点：无 SSR 复杂度，构建产物可直接注入任意网页
缺点：SEO 差，首屏无 SSR 预渲染
扩展：WXT 框架简化 Chrome 扩展 + React + TS 配置

优点：Table、Form、Tree 等复杂组件开箱即用
Ant Design X：专为 AI 对话场景的扩展组件库（Bubble、Sender）

配合：shadcn/ui、Headless UI、Radix 都以 Tailwind 为样式层
注意：单独用 Tailwind 构建复杂组件（Modal、Select）需配合 Headless
前端 — 状态管理 / 数据获取
Zustand 推荐
轻量，Agent 会话 / UI 状态
Jotai 原子化
流数据细粒度更新，Suspense 集成
TanStack Query 推荐
服务端异步状态缓存层
Redux Toolkit 企业级
大型团队 / 复杂状态机，DevTools
Zustand —— Agent 会话状态首选
用途：存储 messages[]、currentTask、toolResults、UI 交互状态
优点：零 boilerplate，TS 泛型友好，支持 immer 中间件
持久化：persist 中间件 + localStorage 做会话恢复
搭配：TanStack Query 管服务端状态，Zustand 管 UI 状态，职责分离
Jotai —— 流式场景细粒度更新
优点：atom 粒度极细，token 逐字渲染只重渲染变化 atom
Suspense：异步 atom 天然集成，loading 状态声明式

Celery —— Python 生产级任务队列
优势
+生态最成熟，文档丰富，支持定时任务（beat）

Build vs Buy —— 先判断你的问题是不是“需要自己造 Agent”
适合买：客服、销售、内训、外呼、法律辅助这类已有成熟 vertical 的赛道
适合自建：你有强业务流程、内部工具、私有数据和差异化体验需求
关键问题：你的壁垒在模型和 orchestration，还是在业务数据和集成？
经验：很多团队其实不需要造平台，只需要买 product + 做轻集成
客服 / 销售 Agent —— 最先成熟的平台化场景
原因：流程标准化、ROI 明确、可以直接量化替代人力与转化率
技术重点：语音、CRM 集成、知识库、工具调用、审计留痕
启发：如果你也做这类场景，重点不在 LLM 调一下就行，而在流程与系统接入
参考玩家：图里出现的 Decagon、Sierra 更像“应用层产品”而非底层框架
法律 / 专业工作流 —— 高准确性和审计要求
特点：长文档、多轮引用、检索精准性和输出可追溯性要求高
技术要求：文档解析、citation、权限控制、审计、人工复核流
启发：这类场景说明“可追溯 + 工作流”往往比“模型聪明”更重要
参考：Harvey 一类产品体现的是 workflow depth，不是简单聊天机器人
个人执行助理 —— 工具链比模型更重要
依赖：邮箱、日历、浏览器、搜索、表单、任务系统等工具接入能力
关键：记忆、权限、浏览器自动化、长任务编排比 prompt 更关键
风险：一旦跨多个 SaaS 执行动作，安全和幂等性问题会急剧上升
参考：Lindy、MultiOn 这类产品更像“自动化平台 + Agent 交互层”

适合：需要 persistent agent、长期 persona、长期用户上下文的产品
和 workflow 区别：它更像“有状态 agent runtime”，不是纯流程编排器

Bedrock Agents —— 云厂商托管式 Agent 平台
优点：和云上 IAM、日志、知识库、模型服务集成紧
适合：AWS 重度用户，平台治理和合规优先的企业环境
限制：可移植性弱，抽象深，调试和迁移成本较高
判断：如果你主要价值在业务，不想自己搭 runtime，可考虑托管型
LiveKit Agents —— 实时交互型 Agent 的运行平台
价值：语音、视频、房间、媒体转发和 Agent 调用统一到一层
适合：会议助手、电话机器人、实时语音 copilot
重点：当媒体链路是核心，不要把它当成普通文本 Agent 来做
结论：实时 Agent 更像“媒体系统 + 智能层”，不是单一模型 API

适合：想快速把 memory 做成现成组件，而不是自己从零设计 schema

mem0 —— 轻量、产品导向的记忆抽取

工程价值：记忆从“prompt 拼接”进化为“工作流节点”
建议：只有在 workflow 已经复杂时再引入，不然会过度设计

适合：网站知识库、竞品监控、外部情报采集
价值：让工具调用聚焦业务，而不是处理网页脏数据
限制：受站点反爬、登录态、动态渲染影响时仍需要浏览器层配合

E2B —— 为 Agent 设计的执行沙箱
适合：代码解释器、数据分析、文件转换、短时工具执行
优点：接入快，抽象贴近 Agent 场景，不必自己从 K8s 开始搭
关键：沙箱生命周期、资源上限、网络权限要单独配置
结论：当 Agent 能写代码时，沙箱不是可选项
Modal —— 更像弹性计算层，不只是沙箱
强项：GPU / CPU 任务、批处理、容器函数、模型服务
适合：需要把执行环境和计算资源一起托管的 Agent 工具
区别：它更接近“运行平台”，安全隔离只是其中一部分
判断：如果任务更像 compute job，Modal 比纯 sandbox 更合适
Firecracker / microVM —— 高隔离执行底座
优势：隔离强度高于普通容器，适合高风险代码执行
适合：政企、金融、用户代码执行、多租户不可信负载
代价：自建复杂度高，调度和镜像管理成本显著上升
建议：安全要求高时才上，不要为了“听起来高级”硬上
Browser Sandbox —— 浏览器自动化也必须隔离
原因：浏览器 Agent 会触网、登录、下载、执行脚本，风险不低于代码执行
适合：网页操作 Agent、RPA、表单填写、站点巡检
建议：浏览器会话、代理、下载目录、凭证注入都要隔离管理
经验：很多团队只做代码沙箱，不做浏览器隔离，这是常见漏洞

LangGraph —— 有状态 Agent 图
特点：图节点 = Agent 步骤，边 = 条件转移，支持循环
状态：内置 checkpoint，每步持久化，崩溃可恢复
场景：ReAct loop、规划-执行-反思循环、Human-in-the-loop
部署：LangGraph Cloud 托管，或自部署
推荐：复杂多步骤 Agent 工作流首选，比手写状态机省 80% 代码。
LlamaIndex —— RAG 专精
强项：数据接入（PDF/网页/数据库）、分块、向量化、检索、重排
高级 RAG：HyDE、父子分块、多向量检索开箱即用
Agent：QueryEngine 作为 Agent 工具，数据驱动 Agent
vs LangChain：RAG 能力更专精，但 Agent 编排弱于 LangGraph
CrewAI —— 角色型多 Agent
概念：Agent 有角色（Researcher / Writer / Reviewer），任务分工
优点：声明式配置，快速搭建多 Agent 协作原型
缺点：复杂流程控制能力弱于 LangGraph
适合：内容生成、研究报告、流水线型多 Agent 任务
AutoGen —— 对话型多 Agent
特点：Agent 间通过自然语言对话协作，代码执行沙箱内置
场景：数据分析 Agent、编程辅助 Agent、研究助手
v0.4：重构为事件驱动架构，生产可用性提升
缺点：多 Agent 对话成本高，token 消耗大
Vercel AI SDK —— TypeScript 全栈 Agent
特点：前端 React Hooks + 后端 Node 统一 API，零 schema 重复
工具：tool() 声明即类型，Zod 验证入参，流式工具结果
流式 UI：streamUI() 可流式返回 React 组件，非仅文本
适合：Next.js + Node 的 TS 全栈 Agent 产品
Pydantic AI —— 类型安全 Agent
优点：与 FastAPI / Pydantic 栈无缝集成，工具 schema 自动推导
类型：Agent 返回值有强类型，减少 LLM 输出解析错误
测试：内置 TestModel，无需调用真实 LLM 可单测 Agent 逻辑
新兴：2024 年底发布，生态尚在成长，生产前评估稳定性

OpenTelemetry —— Agent 与系统观测的公共语言
价值：把模型调用、tool span、HTTP span、DB span 串成同一条 trace
好处：框架可替换，Langfuse / Phoenix / Tempo 都能吃 OTel 数据
建议：从一开始就统一 trace_id，避免后面补链路成本极高
落地：前端、BFF、Agent worker、DB 网关都要传上下文
Prometheus —— 系统与业务指标基座
看什么：QPS、P95、worker backlog、LLM 429、tool 失败率、token 吞吐
模式：pull 抓取 + Alertmanager 告警，适合 K8s / 容器环境
Agent 补充：自定义业务指标必须单独暴露，不能只靠日志
限制：高基数 label 会把时序库打爆，trace_id 不能直接当 label
Grafana —— 把模型成本、系统指标和业务质量放进同一张图
统一视图：把 token 消耗、latency、error rate、队列 backlog 和 RAG 质量并排看
适合：研发、运维、产品共用的跨角色看板
推荐：把“模型成本 / 每任务成功率 / 平均步骤数”做成固定 dashboard
注意：Grafana 负责展示，不负责存储，要和下层系统搭配
Loki / Tempo —— 开源可观测组合
Loki：适合结构化日志，成本低于 Elasticsearch，Grafana 原生兼容
Tempo：专做 trace，和 OTel / Grafana 配合顺畅
适合：想自托管 observability stack，又不想上太重 ELK 的团队
经验：日志里一定要带 trace_id / user_id / task_id，排查才够快

vLLM —— 开源模型服务事实标准
核心：PagedAttention 提升 KV cache 利用率，多并发吞吐显著高于传统 HF serving
场景：Qwen、Llama、DeepSeek 等开源模型对外提供 OpenAI 兼容接口
优势：适合长上下文、多请求 batch、Agent 高并发调用
注意：模型切换和多 LoRA 管理复杂度较高，需单独规划 GPU 池
SGLang —— 复杂 Agent 生成控制更强
强项：结构化解码、speculative decoding、tool-heavy workflow 的控制力更强
适合：代码生成、JSON 输出、需要高可控推理流程的服务
定位：比 vLLM 更偏“推理编程框架”，不是纯通用 serving 替代
建议：当输出格式约束很强时优先评估 SGLang
Ollama —— 本地开发与 PoC 最顺手
优点：一条命令拉模型，Mac / Linux 本地体验非常顺
适合：原型验证、个人工具、本地 copilot、CI 里的轻量冒烟测试
限制：吞吐和多租户能力远弱于 vLLM / KServe 生产方案
结论：开发环境优先，生产环境谨慎
llama.cpp —— CPU / 边缘设备推理首选
价值：量化模型在 CPU、M 系芯片、嵌入式设备上都能跑
场景：本地私有 Agent、离线助手、终端工具、边缘盒子
优点：部署面最广，生态成熟，很多上层产品都基于它
限制：吞吐较低，不适合服务化多用户高并发

LiteLLM —— 多模型代理与成本治理层
价值：把 OpenAI、Anthropic、Gemini、vLLM、自建模型统一成同一接口
功能：fallback、weighted routing、budget、缓存、日志
场景：Agent 按任务类型切模型，或做云上 / 自建混合路由
当前主流：OpenAI 看 GPT-5.4 / GPT-5.4-mini / gpt-realtime-1.5；Claude Code 看 Opus 4.6 / Sonnet 4.6
开源 / 国产：Qwen 看 Qwen3 / Qwen3-Coder / Qwen2.5-VL，GLM 看 GLM-5 / GLM-4.5V，DeepSeek API 当前是 deepseek-chat / deepseek-reasoner（V3.2）
KServe —— 平台化模型托管
适合：已经有 Kubernetes 平台团队，需要统一模型部署规范
能力：autoscaling、canary、GPU 调度、推理服务 CRD
优点：把模型服务纳入平台治理，比各团队自己起容器更规范
成本：系统较重，小团队原型期不必上来就用
BentoML —— Python 推理服务化很顺手
定位：把 Python 模型逻辑、依赖和 API 一起打包成可部署单元
适合：embedding service、reranker、classifier、vision service
好处：研发用 Python 写，交付给平台时仍然像标准服务
边界：大规模集群治理仍需结合 K8s / service mesh
Ray Serve —— 多阶段推理 pipeline
强项：多个模型节点串成服务图，适合复杂 inference pipeline
场景：ASR → VLM → reranker → LLM 这类链式推理服务
适合：GPU 集群较大、模型和数据处理算子很多的平台
缺点：运维复杂度高，单一模型服务时不如 LiteLLM / BentoML 轻量

Whisper：高质量语音转文本，适合离线批处理和轻实时
WhisperX：补齐对齐和 diarization，更适合会议记录、客服质检

MinerU —— 文档理解比传统 OCR 更适合知识库
强项：识别章节、表格、公式、脚注等版面语义，不只是抠字

Unstructured —— 多源文档接入层
作用：把 PDF、网页、Word、邮件等异构文档统一转成可切分元素
优势：接入器多，适合做企业知识库 ingestion pipeline
建议：和自定义清洗、chunking、metadata enrichment 结合使用

常见问题 & 对策
模型层 / 推理层
模型调用、RAG、策略约束和推理稳定性问题
LLM API 限流（TPM / RPM 超限）
现象：大量 429 Too Many Requests，任务失败。对策：令牌桶限流 在请求前扣减 token 预算，超限时入等待队列；多 Key 轮询 分散到多个 API key；指数退避重试 + 抖动；预估 token 用量做调用前预算控制。
RAG 召回质量低 / 幻觉
现象：向量检索结果不准，LLM 基于错误 context 生成幻觉。对策：混合检索（向量 + BM25），Rerank 模型 二次排序（Cohere / BGE）；优化分块策略（父子分块 / 滑动窗口）；在 metadata 上加过滤缩小检索空间；引入 RAG 评估（Ragas）持续监控。
业务策略与模型输出冲突
现象：模型给出“看起来合理”的建议，但和业务规则、风控策略、运营策略冲突，比如不该给优惠时给了优惠。对策：策略先行，先由规则引擎裁边界，再让模型在可选范围内生成；把模型定位成解释器 / 建议器，不是最终裁决者；关键结论必须附带命中的规则或来源。
工具层 / 执行层
工具调用、权限、执行一致性和人工接管问题
工具调用幂等性 & 重复执行
现象：网络重试导致工具（发邮件、写 DB）被执行多次。对策：每次工具调用生成 idempotency_key（UUID 绑定 task+step），服务端 Redis SET NX 去重；工具执行结果缓存，相同参数直接复用 cached 结果。
工具权限漂移 / 越权调用
现象：Agent 随着工具越来越多，出现“本不该调用的工具被调用”“读到了不该读的数据”。对策：最小权限 默认拒绝，按角色 / 场景开放工具；用 Casbin / OPA 管策略；tool schema 增加作用域与敏感级别；高风险动作（退款、改价、发信）必须二次确认或人工审批。
人工接管链路断裂
现象：Agent 无法处理时虽然转人工了，但上下文、工具执行记录、用户情绪和历史状态没有一起交接。对策：handoff packet 标准化，至少包含摘要、最近动作、失败原因、用户意图、工具结果；客服工作台直接展示 trace；转人工时冻结自动回复，防止人与 Agent 同时回复。
数据层 / 记忆层
上下文、状态、价格信息和存储稳定性问题
上下文窗口膨胀 & 内存泄漏
现象：多轮对话后 context 越来越大，延迟线性增加。对策：滑动窗口压缩，超过阈值时摘要旧消息保留重点；Redis TTL 管理会话生命周期；定期 GC 孤立会话对象；对话历史分页加载不全量传入。
数据库连接数耗尽
现象：高并发下 PG 报 "too many connections"。对策：PgBouncer（transaction 模式）连接池，最大连接数限 CPU×4 以内；读多写少加 只读副本 分流；慢查询必须加索引；Python asyncpg 异步驱动替代同步 psycopg2。
价格 / 库存 / 可用性信息过期
现象：电商、酒旅、打车等场景中，Agent 给出的价格、库存、运力或房态已经失效，用户看到的结果与真实交易结果不一致。对策：把搜索与交易分层，展示层可缓存，交易层必须实时查；对高时效字段加 TTL 与版本号；提交前强制二次确认；所有报价类回答附带“刷新时间 / 数据来源”。
记忆污染 / 用户画像写错
现象：Agent 把一次偶然对话写成长期事实，后续反复引用错误偏好或错误身份信息。对策：记忆分层（短期摘要 / 长期事实 / 可删除偏好）；记忆写入必须带置信度与来源；关键记忆支持人工编辑与撤销；对高敏字段先不自动写入，走审核或弱引用模式。
交互层 / 实时链路
用户连接、实时反馈、语音和长任务体验问题
长时任务超时 / 连接中断
现象：Agent 执行 30s+ 导致 HTTP 超时、前端断连。对策：Celery / ARQ 异步化，颁发 task_id；前端用 SSE 轮询 任务状态；WebSocket 保持长连接；支持断线重连续看进度。
实时语音链路卡顿 / 打断失败
现象：生活服务、打车、客服等语音场景里，用户打断无效、延迟过高、TTS 还在说旧内容。对策：VAD、中断信号与音频缓冲必须分层处理；推理、ASR、TTS 走独立 pipeline；常用回复做短语缓存；网络抖动时允许降级到文字或半双工模式。
运营层 / 发布层
版本、渠道一致性、实验和运营治理问题
Agent 执行不可观测 / 调试困难
现象：Agent 出错难以定位是哪步工具调用出了问题。对策：接入 Langfuse / Phoenix 记录每步 Span；结构化日志（JSON）+ 全局 trace_id 串联；每步工具调用记录 input/output/latency/cost；生产环境采样记录，开发环境全量记录。
多渠道答案不一致
现象：同一个问题在 App、客服台、电话、商家后台里回答不同，用户和运营都不信任系统。对策：知识源统一，Prompt 与政策内容版本化；把 FAQ、政策、价格说明、话术从“页面里复制”升级为共享配置中心；发布时做渠道一致性回归测试。
A/B 实验和 Prompt 版本失控
现象：同一业务同时跑多个 Prompt、模型和工具版本，却没有清晰记录，导致效果回溯困难。对策：Prompt 版本化、模型版本化、工具版本化统一挂到 trace；实验流量走明确分桶；上线前后固定看 成功率 / 转化率 / 成本 / 人工接管率 四类指标，不只看主观效果。
合规层 / 安全层
内容审核、监管规则、敏感信息和发布安全问题
内容审核 / 合规拦截不完整
现象：内容创作、电商、客服场景中，模型输出或工具结果触碰版权、广告法、医疗金融合规或平台审核规则。对策：多层审核：输入前、生成后、发布前分别检查；文本、图片、音频分开审；高风险行业引入 规则引擎 + 人工复核，不要只靠单一审核模型。
